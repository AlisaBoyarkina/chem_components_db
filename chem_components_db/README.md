# Django Chemical Components Database

Веб-сервис для управления справочником химических компонентов. Часть учебного проекта по интеграции с Aspen Plus / HYSYS.

## Возможности

- ✅ Хранение информации о химических веществах (название, формула, молярная масса, критические параметры)
- ✅ Синонимы компонентов (русские, английские, Aspen Plus, HYSYS)
- ✅ Физико-химические свойства компонентов
- ✅ Django Admin для управления данными
- ✅ Источники данных (NIST, справочники, ручной ввод)
- ✅ Логирование импорта данных из CSV/Excel

## Быстрый старт

### 1. Клонировать проект

```bash
git clone https://github.com/AlisaBoyarkina/chem_components_db.git
cd chem_components_db
```

### 2. Создать виртуальное окружение

**На macOS / Linux:**
```bash
python3 -m venv venv
source venv/bin/activate
```

**На Windows (CMD):**
```bash
python -m venv venv
venv\Scripts\activate
```

### 3. Установить зависимости

```bash
pip install -r requirements.txt
```

### 4. Применить миграции (создать БД)

```bash
python3 manage.py migrate
```

### 5. Создать суперпользователя (администратор)

```bash
python3 manage.py createsuperuser
```

Введи:
- Username: `admin`
- Email: `test@test.com`
- Password: любой пароль

### 6. Запустить сервер

```bash
python3 manage.py runserver
```

Сервер запустится на `http://127.0.0.1:8000/`

### 7. Зайти в админ-панель

Открой браузер: `http://127.0.0.1:8000/admin/`

Введи учётные данные суперпользователя.

## Модели БД

### Component
Основная таблица с информацией о химических веществах.

**Поля:** name, formula, cas_number, molar_mass, substance_class, normal_boiling_point, critical_temperature, critical_pressure, acentric_factor

### ComponentAlias
Синонимы компонентов (разные названия в разных системах).

**Типы:** ru (русское), en (английское), aspen (Aspen Plus), hysys (HYSYS)

### Property
Физико-химические свойства компонентов.

**Поля:** property_name, value, unit, conditions, source (DataSource)

### DataSource
Источники данных (NIST, справочники, ручной ввод).

### ImportLog
Журнал импорта данных из CSV/Excel.

## 📑 Архитектурные пояснения к реализации REST API
Разработка программного интерфейса (API) для базы данных химических компонентов опирается на строгие архитектурные принципы. Ниже приведено детальное теоретическое и техническое обоснование каждого выбранного решения.
------------------------------
## 1. Выбор архитектурного паттерна: DRF ViewSets и Routers
Вместо классических представлений на базе функций (@api_view) или классов (APIView), в проекте использован связный паттерн ReadOnlyModelViewSet совместно с DefaultRouter.

* Исключение дублирования кода (DRY): Один класс ComponentViewSet автоматически генерирует обработку двух базовых сценариев: вывод упорядоченного списка сущностей (эндпоинт /components/) и чтение конкретной записи по её первичному ключу (/components/{id}/).
* Стандартизация URL: Роутер самостоятельно формирует URL-пути, следит за наличием или отсутствием замыкающих косых черт (trailing slashes) и настраивает корректные имена маршрутов для генерации ссылок внутри системы.
* Безопасность данных: Класс ReadOnlyModelViewSet жестко ограничивает доступные HTTP-методы только безопасными операциями: GET, HEAD и OPTIONS. Внешние скрипты физически не смогут деструктивно повлиять на базу данных (вызвать POST, PUT или DELETE). Это гарантирует целостность справочника.

------------------------------
## 2. Динамическое разделение сериализаторов (List vs Detail)
Для оптимизации сетевой нагрузки и ускорения работы интерфейсов в коде переопределен метод get_serializer_class(). Система динамически выбирает, в каком объеме отдавать данные:

* ComponentListSerializer (Для списков): Содержит только базовые идентификаторы (название, CAS, формула, класс). Он используется, когда внешний скрипт или фронтенд запрашивают общую выборку. Это позволяет не перегружать сеть тяжелыми текстовыми массивами и таблицами свойств, когда пользователю нужно просто пролистать перечень веществ.
* ComponentDetailSerializer (Для карточки вещества): Включает в себя полные фундаментальные константы (критические точки, ацентрический фактор), а также вложенные структуры синонимов и теплофизических свойств. Он вызывается точечно, когда система уверена, что пользователю необходим полный физико-химический паспорт конкретного компонента.

------------------------------
## 3. Решение проблемы производительности SQL (N+1 Query Problem)
При работе с реляционными базами данных (SQLite/PostgreSQL) частой ошибкой является наивный вызов связанных данных. Если у 20 компонентов запросить их свойства и синонимы, стандартный ORM Django выполнит 1 начальный запрос и по 2 дополнительных запроса для каждого компонента (итого $1 + 20 \times 2 = 41$ обращение к БД).

* Решение: В коде вьюсета применена конструкция .prefetch_related('componentalias_set', 'property_set__source').
* Механизм работы: Django выполняет ровно 3 комплексных SQL-запроса на уровне базы данных: первый извлекает компоненты, второй — все связанные синонимы для этих компонентов, третий — свойства вместе с именами их источников (DataSource). Затем ORM самостоятельно "склеивает" эти массивы в оперативной памяти Python. Нагрузка на базу данных снижается до минимума, а скорость отдачи JSON возрастает в разы.

------------------------------
## 4. Реализация сквозного гибкого поиска
Параметр search_fields настроен таким образом, чтобы покрыть любые сценарии использования инженерами-технологами:

* Поиск по точным индексам: Интеграция по cas_number позволяет расчетным программам осуществлять абсолютно точную идентификацию вещества (так как CAS уникален в международной практике).
* Поиск по бэкграунду симуляторов: Связка componentalias__alias_name делает API "умным". Если внешняя программа запрашивает компонент по его внутреннему имени из Aspen Plus (например, METHANE капсом), поисковый движок Django заглядывает в связанную таблицу синонимов, находит совпадение и возвращает верный компонент, даже если в основном поле name он записан как "Метан" или "Methane".
