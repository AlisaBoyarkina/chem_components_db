## 2. Основные задачи и результаты

### Задача 1: Проектирование моделей Django
**Статус:** ✅ Выполнено

Спроектированы 5 моделей (классов) для представления структуры данных:

1. **Component** — основная таблица с информацией о веществах
   - Поля: название, формула, CAS-номер, молярная масса
   - Дополнительно: класс вещества, критические параметры (температура, давление), ацентрический фактор
   - Связи: 1 компонент → много синонимов, много свойств

2. **ComponentAlias** — синонимы компонентов (разные названия в разных системах)
   - Типы: русское, английское, Aspen Plus, HYSYS
   - Назначение: позволяет искать компонент по названию в любой системе

3. **Property** — физико-химические свойства компонентов
   - Данные: название свойства, значение, единица, условия
   - Привязка: к компоненту и источнику данных

4. **DataSource** — справочник источников данных
   - Данные: название, описание, веб-ссылка
   - Использование: отслеживание происхождения информации

5. **ImportLog** — журнал импорта файлов
   - Назначение: логирование результатов загрузки CSV/Excel
   - Содержит: имя файла, количество успехов/ошибок, детали ошибок (JSON)

**Файл кода:** `components/models.py` (см. приложение)

### Задача 2: Настройка связей между моделями
**Статус:** ✅ Выполнено

- **Component ↔ ComponentAlias:** ForeignKey с `on_delete=CASCADE` (если удалить компонент, удалятся все синонимы)
- **Component ↔ Property:** ForeignKey с `on_delete=CASCADE` (если удалить компонент, удалятся свойства)
- **DataSource ↔ Property:** ForeignKey с `on_delete=SET_NULL` (если удалить источник, свойства останутся)
- **ImportLog:** независимая таблица (журнал, не связан с другими FK)

**Ограничения целостности:**
- Все обязательные поля помечены как `null=False, blank=False`
- Опциональные поля — `null=True, blank=True`
- CAS-номер рекомендуется уникальным (но может быть пусто)

## 3. Описание реализованной структуры БД

### Таблицы и их назначение

```
Component (основная)
├── id (PK)
├── name, formula, cas_number
├── molar_mass
├── substance_class
├── критические параметры (critical_temp, critical_press, acentric_factor)
└── normal_boiling_point

    ↓ 1:M
    
ComponentAlias (синонимы)
├── id (PK)
├── component_id (FK → Component)
├── alias_name (текст)
└── alias_type (выбор: ru/en/aspen/hysys)

Property (свойства)
├── id (PK)
├── component_id (FK → Component)
├── property_name
├── value, unit, conditions
└── source_id (FK → DataSource, опционально)

DataSource (источники)
├── id (PK)
├── name
├── description
└── url

ImportLog (журнал)
├── id (PK)
├── imported_at (авто)
├── file_name, total_rows, success_count, error_count
└── errors (JSON)
```

### Логика работы

1. **Ввод данных:** Через админ-панель добавляются компоненты, синонимы, свойства, источники
2. **Поиск:** Система может найти компонент по названию в любой форме (русское, английское, Aspen/HYSYS)
3. **Интеграция:** API (участник 4) использует таблицы для выдачи JSON-ответов второй команде (Aspen/HYSYS)
4. **Импорт:** Файлы CSV/Excel парсятся (участник 3) и загружаются в Component/Property, результат записывается в ImportLog

## 8. Трудности и решения

| Проблема | Решение |
|----------|---------|
| Опечатки при переписывании кода вручную | Внимательная проверка, использование IDE с подсветкой синтаксиса |
| Неправильные отступы (IndentationError) | Выравнивание всех методов на одном уровне с полями класса |
| Не добавлено приложение в INSTALLED_APPS | Добавление 'components' в config/settings.py |
| Забыта регистрация модели в admin.py | Добавление всех 5 моделей через admin.site.register() |
